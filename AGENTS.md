# 项目维护规则

## 版本号

- 每个包含源码、界面、脚本、配置、文档或资源修改的交付批次，都必须在提交和推送前提升一次语义化版本号。
- 新增向后兼容的功能提升次版本号，向后兼容的问题修复提升修订号，不兼容变更提升主版本号。
- 版本号必须同步到 `Cargo.toml`、根包在 `Cargo.lock` 中的版本、`package.json`、`package-lock.json` 和 `tauri.conf.json`。
- 界面版本从 `package.json` 读取，不得再次硬编码；README 中带版本号的产物路径必须同步更新。
- `scripts/Build-Release.ps1` 必须在构建前校验所有版本来源一致。版本不一致时不得构建、提交或发布。
- 同一批连续修改只提升一次版本号；开始下一批修改时再次提升。

## 中文与编码

- 所有文本文件使用 UTF-8。PowerShell 读取和写入中文时必须显式指定 UTF-8。
- 中文界面必须继续使用项目内的 Noto Sans CJK SC 字体兜底，并在正式构建中实际检查乱码、方框、截断和重叠。
- 中文动态消息、日志和 JSON/TOML 配置必须保持 UTF-8 边界。

## 交付检查

- 至少运行 TypeScript 检查、前端生产构建、Rust 测试、`cargo clippy --all-targets -- -D warnings` 和 Tauri Release 构建。
- 发布前运行 `npm audit` 和 `cargo audit`；不得带着可修复的已知漏洞发布，目标平台之外的维护警告必须确认依赖链和影响范围。
- 正式版必须实际检查窗口无意外滚动条、无黑色控制台窗口、中文显示正常，主要按钮和弹窗可操作。
- 发布前检查 Git 状态、敏感信息、安装包与便携包内容，并确认 GitHub Release 标签指向默认分支最新提交。
